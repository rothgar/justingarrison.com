This build took much longer than I anticipated.
Partially because I spent a lot of time trying things that didn't work out, partially because I had to learn new skills, and mostly because I didn't have a deadline.

I've been running it as part of my home lab with [Talos Linux](https://talos.dev) for the past couple of months without any hardware failures so I'm sharing it with you.
Watch the video for a lot more pictures or keep reading to learn about the parts and software.

## Build planning

If you watch the video you'll see all the things I did wrong with this build and plans that didn't work out.
I learned about PCB design and printing and a lot of limits with power distribution and small spaces.

Even though every component I was making was **larger** than its default case, it was still a fun challenge to cram my components in these cases.

My first challenge was to collect all of the cases I wanted to use.
Once I figured out I wanted 5 standard size Linksys cases and 1 fat case I scoured thrift stores, asked friends, and bought 1 on ebay.

![a stack of linksys routers](/img/wrtk8s-full-stack.jpg)

Putting together all of the cases I had accumulated was encouraging but multiple setbacks kept slowing me down.
Designing and ordering PCBs, soldering USB-C connectors, and general design decisions made this project move slowly.
I also wasn't in desperate need for more compute in my home lab so there was no hurry.

I wanted to add a storage component to the cluster to replace my Synology, but storage prices are too high and I ran out of Linksys cases.
If you have a Linksys case you would like to donate, please [reach out](mailto:justin@linux.com).

## Case Build

Each case had its own challenges.
The only one that was straightforward was the base switch layer.
Once I removed the old case on the switch it mounted without any issues.

The Raspberry Pi cases needed some work because with the PoE HATs and base mounting plate they didn't quite fit.
I had to trim the HAT pins and on the Pi 5s I couldn't use the base mounting plate.
That's why they look a little crooked from the rear.

![a stack of linksys cases with HDMI cables coming out](/img/wrtk8s-pi-hdmi.jpg)

I had also debated leaving HDMI connections on each Pi just to make it easier to debug things when they went wrong.
I hated how it looked so I ended up leaving them off and whenever I need to debug something I can flash a new SD card or open the case and plug in video.

The LED status lights really makes the stack look more complete.
I only did this for the Raspberry pis, but it's still enough activity IMO.

It took far too long to try and get the config.txt to work for both devices and activity LEDs.
It turns out Raspberry Pis don't have a way to trigger activity for network or _any_ storage.
My config ended up with this, which just powers on the red LED when the system is on.

```text
# pi 5
dtparam=pwr_led_gpio=26
dtparam=pwr_led_activelow=0
dtparam=pwr_led_trigger=default-on

# pi 4
gpio=26=op,dh
```

I then run a daemonset targetting all Raspberry Pis with a [vibe coded python script](https://github.com/rothgar/homelab-flux/blob/main/clusters/wrtk8s/rpi-led-daemon/configmap.yaml) that blinks the green LED based on network traffic.

The mini PC fit without any problems. I only needed to figure out how to extend the power and reset buttons and give an external oculink connection.
The power buttons took me longer to figure out than I expected.
Soldering to the buttons worked but would stop working after a few presses so I knew I needed to switch to something mechanical.

![two rocker buttons on a pink build plate](/img/wrtk8s-mini-buttons.jpg)

It took me a few iterations on how to mount them and exactly what length I needed, but once I got it working they've worked well.

The eGPU power supply and GPU took a lot more cutting than I initially expected, but overall I'm happy with the result.
The top case is the one I broke but it came from my original "hot rod" so the case already had a large hole in the top.

It would have been nice if I could get a smaller power supply to power the GPU that didn't need to stick out of the side, but I didn't find anything that would put out enough power.
If you know of something, please let me know.

## Usage

![a grafana dashboard showing availability of services](/img/wrtk8s-grafana.png)

The mini PC runs the control plane components and Talos and Kubernetes upgrades are managed via [Talos Omni](https://www.siderolabs.com/omni).
The entire cluster is managed with a [cluster template](https://github.com/rothgar/homelab-flux/blob/main/clusters/wrtk8s/cluster.yaml) that defines all of the machines, patches, and configuration.

It has plenty of capacity for my small cluster.
The total cluster includes:

- GMKtec NucBox M6 Mini with Nvidia RTX 2060
- 2x Pi 4 8gb
- 2x Pi 5 8gb
- HP Z4 G4 mini PC with Nvidia P1000
- [LattePanda IOTA](https://www.lattepanda.com/lattepanda-iota)
- [NVIDIA DGX Spark](https://www.nvidia.com/en-us/products/workstations/dgx-spark/)

Storage for the cluster is provided by a Synology DS718+ which previously was running all of my workloads by itself and was too slow to keep up.
I still use it for storage via the [Synology CSI driver](https://github.com/SynologyOpenSource/synology-csi).

![a messy closet with cables, fans, and computers](/img/wrtk8s-other-infra.jpg)

Only the top half of that list is actually in the Linksys cases.
If I get the PoE adapter working for the IOTA I'll move it into one of the spare Linksys cases, but it's going to need more planning because it's too tall to fit inside the case.

Each Raspberry Pi runs Talos from the microSD card and has secondary storage (NVMe on Pi 5 and USB on Pi 4) for additional data.
The USB drives have been very slow and I'm thinking about replacing them, but I haven't found an alternative option yet.

Kubernetes on the hardware has been rock solid and I'm sad I didn't do this sooner.
I know I've built a lot of clusters in the past, but I had always avoided running my home services on Kubernetes for complexity reasons.

Even with 3 GPUs in the cluster I run into GPU scheduling constraints more than anything else.
Occasionally I'll find a workload that doesn't have arm builds of containers, but that's a simple thing to build and push myself.

I'll have a follow-up blog post and video to talk about what I run inside the cluster.
